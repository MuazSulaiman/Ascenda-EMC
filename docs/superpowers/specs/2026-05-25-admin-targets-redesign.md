# Admin Targets — Redesign Spec
**Date:** 2026-05-25
**Status:** Approved for implementation
**Replaces:** `docs/superpowers/specs/2026-05-04-target-management-design.md`

---

## 1. Why This Redesign

The original module had three fundamental UX problems:

1. **Three layers of manual entry.** Admin had to enter the same numbers at year level, rep level, and breakdown level. Redundant and error-prone.
2. **Four disconnected tabs.** No sense of workflow or context. Switching between Rep Allocation and Rep Breakdown lost context constantly.
3. **Exhausting breakdown form.** Five cascading dropdowns + auto-level detection + override toggle to add a single row.

This redesign fixes all three: bottom-up data entry (breakdown rows are the source of truth), a split-panel layout (no tab switching), and a level-first form (pick level → one relevant dropdown appears).

---

## 2. Architecture Decisions

### 2.1 Bottom-Up Hybrid Model
The year budget is manually set — it is handed down from management and serves as the planning ceiling. Everything below it is derived:

```
target_year.budget_amount    ← admin sets (planning ceiling)
target_rep                   ← stores year-user link only (no amounts)
target_breakdown.target_amount ← admin sets (bottom-up entry)

Rep planned total  = SUM(target_breakdown.target_amount) per rep
Year planned total = SUM of all rep totals
```

The `target_rep` table is a pure junction table. There is no "rep target" to set — a rep's total IS the sum of their breakdown rows.

### 2.2 Clean Slate Schema
All previous target tables (`target_year`, `target_rep`, `target_breakdown`) are dropped and recreated. `product_categories` is retained (it is a shared dimension table).

### 2.3 Level-First Breakdown Entry
Admin selects the breakdown level first (radio button). Only then does the relevant dimension dropdown appear. No cascading chain of five dropdowns. Breakdown level is explicit (chosen by admin), not auto-derived.

Valid levels: `customer`, `business_unit`, `product_category`, `business_line`, `article`

The `rep` pseudo-level from the old design is removed. A rep with no breakdown rows has a planned total of zero.

### 2.4 Split Panel Layout
Single page, two columns. No tabs. Left panel = year list + rep list. Right panel = context-sensitive: year overview when no rep selected, rep breakdown editor when rep selected. Session state preserves selection across reruns.

### 2.5 Duplicate Prevention
Uniqueness enforced at `(target_rep_id, breakdown_level, all dimension ids)`. Including `breakdown_level` in the key means a customer-level row and a business-unit-level row for overlapping data are not considered duplicates. DB-level unique index is the enforcement layer; UI checks before insert for a friendly error message.

### 2.6 Validation Changes from Previous Design
| Rule | Old | New |
|------|-----|-----|
| Rep sum vs rep target | Hard block | Removed (no rep target) |
| Year sum vs year budget | Hard confirmation checkbox | Soft warning banner (not a blocker) |
| Duplicate breakdown combo | Hard block | Hard block (kept) |
| LOCKED year edits | Hard block | Hard block (kept) |
| Zero-value row | Soft warning + checkbox | Soft inline warning (not a blocker) |

## 3. Database Schema

### 3.1 Drop Old Tables
```sql
DROP TABLE IF EXISTS target_breakdown CASCADE;
DROP TABLE IF EXISTS target_rep CASCADE;
DROP TABLE IF EXISTS target_year CASCADE;
```

### 3.2 `target_year`
```sql
CREATE TABLE target_year (
    year           INTEGER       PRIMARY KEY,
    budget_amount  NUMERIC(18,2) NOT NULL DEFAULT 0 CHECK (budget_amount >= 0),
    budget_visits  INTEGER       NOT NULL DEFAULT 0 CHECK (budget_visits >= 0),
    status         TEXT          NOT NULL DEFAULT 'DRAFT'
                   CHECK (status IN ('DRAFT', 'ACTIVE', 'LOCKED')),
    created_at     TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
    created_by     INTEGER       REFERENCES users(user_id),
    updated_at     TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
    updated_by     INTEGER       REFERENCES users(user_id)
);
```

Column renamed `target_amount` → `budget_amount`, `target_visits` → `budget_visits` to clearly signal these are ceilings, not planned values.

### 3.3 `target_rep`
```sql
CREATE TABLE target_rep (
    id         BIGINT      GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    year       INTEGER     NOT NULL REFERENCES target_year(year) ON DELETE CASCADE,
    user_id    INTEGER     NOT NULL REFERENCES users(user_id),
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    created_by INTEGER     REFERENCES users(user_id),
    UNIQUE (year, user_id)
);
```

No `target_amount` or `target_visits` — those were the redundant middle layer. Removed entirely.

### 3.4 `target_breakdown`
```sql
CREATE TABLE target_breakdown (
    id                  BIGINT        GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    target_rep_id       BIGINT        NOT NULL REFERENCES target_rep(id) ON DELETE CASCADE,
    year                INTEGER       NOT NULL,
    user_id             INTEGER       NOT NULL REFERENCES users(user_id),
    breakdown_level     TEXT          NOT NULL
                        CHECK (breakdown_level IN (
                            'customer', 'business_unit', 'product_category',
                            'business_line', 'article'
                        )),
    customer_id         INTEGER       REFERENCES customers(customer_id),
    business_unit_id    INTEGER       REFERENCES business_units(business_unit_id),
    product_category_id INTEGER       REFERENCES product_categories(product_category_id),
    business_line_id    INTEGER       REFERENCES business_lines(business_line_id),
    article_id          TEXT          REFERENCES items(product_id),
    target_amount       NUMERIC(18,2) NOT NULL DEFAULT 0 CHECK (target_amount >= 0),
    target_visits       INTEGER       NOT NULL DEFAULT 0 CHECK (target_visits >= 0),
    created_at          TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
    created_by          INTEGER       REFERENCES users(user_id),
    updated_at          TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
    updated_by          INTEGER       REFERENCES users(user_id)
);
```

`year` and `user_id` are denormalized on each row for Power BI direct access without joining through `target_rep`.

### 3.5 Indexes and Constraints
```sql
CREATE INDEX idx_target_rep_year     ON target_rep(year);
CREATE INDEX idx_target_rep_user     ON target_rep(user_id);
CREATE INDEX idx_target_bd_rep       ON target_breakdown(target_rep_id);
CREATE INDEX idx_target_bd_year_user ON target_breakdown(year, user_id);
CREATE INDEX idx_target_bd_customer  ON target_breakdown(customer_id);
CREATE INDEX idx_target_bd_bu        ON target_breakdown(business_unit_id);
CREATE INDEX idx_target_bd_level     ON target_breakdown(breakdown_level);

-- Uniqueness: one row per rep per level per dimension combination
CREATE UNIQUE INDEX uq_target_breakdown ON target_breakdown (
    target_rep_id,
    breakdown_level,
    COALESCE(customer_id,          -1),
    COALESCE(business_unit_id,     -1),
    COALESCE(product_category_id,  -1),
    COALESCE(business_line_id,     -1),
    COALESCE(article_id,           '')
);
```

## 4. Reporting Views

### 4.1 `v_target_year_summary`
Year-level view for the left panel and Power BI year KPIs.
```sql
CREATE OR REPLACE VIEW v_target_year_summary AS
SELECT
    ty.year,
    ty.budget_amount,
    ty.budget_visits,
    ty.status,
    COUNT(DISTINCT tr.id)                                       AS rep_count,
    COALESCE(SUM(tb.target_amount), 0)                          AS planned_amount,
    COALESCE(SUM(tb.target_visits), 0)                          AS planned_visits,
    ty.budget_amount - COALESCE(SUM(tb.target_amount), 0)       AS remaining_amount,
    ty.budget_visits - COALESCE(SUM(tb.target_visits), 0)       AS remaining_visits,
    CASE WHEN ty.budget_amount > 0
         THEN ROUND(COALESCE(SUM(tb.target_amount), 0) / ty.budget_amount * 100, 1)
         ELSE NULL END                                          AS planned_pct,
    ty.created_at,
    ty.updated_at
FROM target_year ty
LEFT JOIN target_rep tr ON tr.year = ty.year
LEFT JOIN target_breakdown tb ON tb.target_rep_id = tr.id
GROUP BY ty.year, ty.budget_amount, ty.budget_visits,
         ty.status, ty.created_at, ty.updated_at;
```

### 4.2 `v_target_rep_summary`
Rep-level view for the left panel rep list and Power BI rep KPIs.
```sql
CREATE OR REPLACE VIEW v_target_rep_summary AS
SELECT
    tr.id                                                        AS target_rep_id,
    tr.year,
    tr.user_id,
    u.name                                                       AS rep_name,
    u.region                                                     AS rep_region,
    u.role                                                       AS rep_role,
    ty.budget_amount                                             AS year_budget_amount,
    ty.budget_visits                                             AS year_budget_visits,
    ty.status                                                    AS year_status,
    COALESCE(SUM(tb.target_amount), 0)                           AS planned_amount,
    COALESCE(SUM(tb.target_visits), 0)                           AS planned_visits,
    COUNT(tb.id)                                                 AS breakdown_row_count,
    CASE WHEN ty.budget_amount > 0
         THEN ROUND(COALESCE(SUM(tb.target_amount), 0) / ty.budget_amount * 100, 1)
         ELSE NULL END                                           AS pct_of_year_budget
FROM target_rep tr
JOIN users u        ON u.user_id   = tr.user_id
JOIN target_year ty ON ty.year     = tr.year
LEFT JOIN target_breakdown tb ON tb.target_rep_id = tr.id
GROUP BY tr.id, tr.year, tr.user_id, u.name, u.region, u.role,
         ty.budget_amount, ty.budget_visits, ty.status;
```

### 4.3 `v_target_breakdown` (Primary Power BI Source)
Full flat view with all dimension FKs resolved to names. This is the main data source for all Power BI target visuals. Power BI slices by `breakdown_level` to avoid double-counting.
```sql
CREATE OR REPLACE VIEW v_target_breakdown AS
SELECT
    tb.id,
    tb.target_rep_id,
    tb.year,
    tb.breakdown_level,
    tb.user_id,
    u.name                   AS rep_name,
    u.region                 AS rep_region,
    u.role                   AS rep_role,
    ty.budget_amount         AS year_budget_amount,
    ty.budget_visits         AS year_budget_visits,
    ty.status                AS year_status,
    tb.customer_id,
    c.account_name           AS customer_name,
    c.sector                 AS customer_sector,
    c.region                 AS customer_region,
    tb.business_unit_id,
    bu.name                  AS business_unit_name,
    tb.product_category_id,
    pc.name                  AS product_category_name,
    tb.business_line_id,
    bl.name                  AS business_line_name,
    tb.article_id            AS product_id,
    i.article_number,
    i.description            AS article_description,
    tb.target_amount,
    tb.target_visits,
    CASE WHEN tb.target_visits > 0
         THEN ROUND(tb.target_amount / tb.target_visits, 2)
         ELSE NULL END       AS productivity,
    tb.created_at,
    tb.updated_at
FROM target_breakdown tb
JOIN  target_year       ty ON ty.year                    = tb.year
JOIN  users             u  ON u.user_id                  = tb.user_id
LEFT JOIN customers     c  ON c.customer_id              = tb.customer_id
LEFT JOIN business_units bu ON bu.business_unit_id       = tb.business_unit_id
LEFT JOIN product_categories pc ON pc.product_category_id = tb.product_category_id
LEFT JOIN business_lines bl ON bl.business_line_id       = tb.business_line_id
LEFT JOIN items         i  ON i.product_id               = tb.article_id;
```

### 4.4 Level-Specific Actual vs Target Views
One view per breakdown level. Each joins visits only on the dimensions meaningful at that level, eliminating cross-level double-counting. Power BI connects each report page to the appropriate view.

All views expose: all target columns from `v_target_breakdown` + `actual_visits` + `visit_achievement_pct` + `NULL AS actual_sales_amount` (placeholder until sales data is wired).

```sql
-- v_actual_vs_target_customer
CREATE OR REPLACE VIEW v_actual_vs_target_customer AS
SELECT
    vt.*,
    COUNT(DISTINCT v.visit_id)  AS actual_visits,
    CASE WHEN vt.target_visits > 0
         THEN ROUND(COUNT(DISTINCT v.visit_id)::NUMERIC / vt.target_visits * 100, 1)
         ELSE NULL END          AS visit_achievement_pct,
    NULL::NUMERIC               AS actual_sales_amount
FROM v_target_breakdown vt
LEFT JOIN visits v
    ON  v.user_id     = vt.user_id
    AND v.customer_id = vt.customer_id
    AND EXTRACT(YEAR FROM COALESCE(v.submitted_at_local, v.submitted_at_utc)) = vt.year
    AND v.is_deleted  = FALSE
WHERE vt.breakdown_level = 'customer'
GROUP BY vt.id, vt.target_rep_id, vt.year, vt.breakdown_level, vt.user_id,
         vt.rep_name, vt.rep_region, vt.rep_role, vt.year_budget_amount,
         vt.year_budget_visits, vt.year_status, vt.customer_id, vt.customer_name,
         vt.customer_sector, vt.customer_region, vt.business_unit_id,
         vt.business_unit_name, vt.product_category_id, vt.product_category_name,
         vt.business_line_id, vt.business_line_name, vt.product_id,
         vt.article_number, vt.article_description, vt.target_amount,
         vt.target_visits, vt.productivity, vt.created_at, vt.updated_at;

-- (Similar pattern for business_unit, product_category, business_line, article)
-- Each filters by breakdown_level and joins visits on the relevant dimension column.
```

The `business_unit` and `product_category` views join visits via `business_lines`:
- `business_unit`: visits where `business_line_id IN (SELECT id FROM business_lines WHERE business_unit_id = vt.business_unit_id)`
- `product_category`: visits where `business_line_id IN (SELECT id FROM business_lines WHERE product_category_id = vt.product_category_id)`

## 5. UI/UX Design

### 5.1 Page Structure
Single admin page. No tabs. Two-column layout: left (30%) + right (70%).

```
┌────────────────────┬──────────────────────────────────────────────┐
│  LEFT PANEL        │  RIGHT PANEL (3 states)                      │
│                    │                                              │
│  Year list         │  A: No year selected → prompt               │
│  ─────────────     │  B: Year selected, no rep → year overview   │
│  Rep list          │  C: Year + rep selected → breakdown editor  │
│  [+ Add Rep]       │                                              │
└────────────────────┴──────────────────────────────────────────────┘
```

Session state keys: `tgt/year` (selected year int), `tgt/rep_id` (selected target_rep_id int).

### 5.2 Left Panel — Year List
- Each row: `● 2025  ACTIVE` or `○ 2024  LOCKED`
- Colored dot: green = ACTIVE, blue = LOCKED, grey = DRAFT
- Click → sets `tgt/year`, clears `tgt/rep_id`
- Selected year is highlighted
- `[+ New Year]` at bottom → right panel shows year creation form (inline, not a new tab)
- Defaults on load: most recent ACTIVE year, or most recent year if none active

### 5.3 Left Panel — Rep List
Shown after a year is selected. Separated from year list by a divider.

- Each row: rep name + planned amount + row count indicator
  - Green dot: has breakdown rows
  - Grey dot + ⚠: zero breakdown rows
- Click → sets `tgt/rep_id`, loads breakdown editor in right panel
- `[+ Add Rep]` at bottom → right panel shows user selector form
- Year LOCKED: rep list is read-only, `[+ Add Rep]` hidden

### 5.4 Right Panel — State A (No Year Selected)
```
  Select a year from the left, or create one to get started.
  [+ Create First Year]
```

### 5.5 Right Panel — State B (Year Overview)
Shown when a year is selected but no rep is selected.

```
  2025                                    ACTIVE
  Budget:  SAR 1,200,000 · 300 visits
  Planned: SAR 960,000  ████████████████░░░░  80%   240 visits
  
  [Edit Year]   [Activate]  /  [Lock]

  ─────────────────────────────────────────────────────────
  Rep            │ Planned       │ Visits │ Rows │ % Budget
  Ahmed          │ SAR 130,000   │   70   │  2   │  10.8%
  Fatima         │ SAR  89,000   │   50   │  3   │   7.4%
  Omar           │ SAR       0   │    0   │  0   │   0% ⚠
  ─────────────────────────────────────────────────────────
  Total planned  │ SAR 960,000   │  240   │  18  │  80%
```

Year edit form (shown inline when `[Edit Year]` clicked, or for new year creation):
- Fields: Year (int, disabled after creation), Budget Amount (SAR), Budget Visits
- Computed display: Productivity = Budget Amount / Budget Visits (display only)
- Save / Cancel buttons

Status transitions:
- DRAFT → ACTIVE: `[Activate]` button, one click, no confirmation
- ACTIVE → LOCKED: `[Lock]` button, requires typing the year to confirm
- LOCKED: all edit controls hidden, lock state is permanent

### 5.6 Right Panel — State C (Rep Breakdown Editor)
Shown when a year and rep are both selected.

```
  Ahmed Al-Qahtani  ·  Rep Region: Riyadh
  Planned: SAR 130,000 · 70 visits  (10.8% of year budget)
  [Remove Rep from Year]   ← only shown if DRAFT/ACTIVE and unlocked

  ─── Add Row ──────────────────────────────────────────────────────
  Level:
    (●) Customer  ( ) Bus. Unit  ( ) Prod. Category  ( ) Bus. Line  ( ) Article

  Customer: [ Al-Nahdi Medical        ▾ ]
  Amount:   [ 50,000  ]    Visits: [ 30 ]    [ + Add Row ]

  ──────────────────────────────────────────────────────────────────
  Level       │ Dimension           │ Amount    │ Visits │
  customer    │ Al-Nahdi Medical    │ 50,000    │  30    │ [×]
  bus. unit   │ Medical Devices     │ 80,000    │  40    │ [×]
  ──────────────────────────────────────────────────────────────────
  Rep total                         │ 130,000   │  70    │
```

**Add row form behaviour:**
- Level radio selection → only the relevant dropdown is shown (no other dropdowns rendered)
- Dimension dropdowns are filtered: Product Category filtered by BU if BU is known from existing rows; Business Line filtered by Product Category; Article filtered by Business Line
- After `[+ Add Row]`: row appears at bottom of table, rep total updates, form resets to empty

**Delete row:** Click `[×]` → inline confirm message replaces the row: "Delete this row? [Confirm] [Cancel]". No full-page dialog.

**Over-budget warning:** If rep total now exceeds year budget, a yellow banner appears above the table. It is informational only — it does not block the add.

**Remove Rep from Year:** Small link at top of the editor. If rep has rows, shows: "This will delete all N breakdown rows. [Confirm Remove] [Cancel]". If year is LOCKED, link is hidden.

**Year LOCKED state:** All form controls hidden. Table is read-only. A lock badge is shown at the top of the state C panel.

## 6. Business Logic & Validation

### Year Level
| Rule | Type |
|------|------|
| Year must be 2000–2099, unique | Hard block |
| Budget amount and visits ≥ 0 | Hard block |
| LOCKED year — no edits | Hard block |

### Rep Level
| Rule | Type |
|------|------|
| Duplicate user for same year | Hard block (DB UNIQUE enforces) |
| Remove rep with rows | Confirmation required (shows row count) |
| Remove rep in LOCKED year | Hidden/blocked |

### Breakdown Level
| Rule | Type |
|------|------|
| Duplicate: same rep + level + dimension | Hard block with inline error |
| Amount and visits both zero | Soft warning (not a blocker) |
| Rep planned total exceeds year budget | Soft warning banner (not a blocker) |
| LOCKED year — no add/delete | Form hidden |
| Delete row | Inline confirmation (Confirm / Cancel) |

### Dimension Validation per Level
| Level | Required dimension field |
|-------|--------------------------|
| customer | `customer_id` must not be null |
| business_unit | `business_unit_id` must not be null |
| product_category | `product_category_id` must not be null |
| business_line | `business_line_id` must not be null |
| article | `article_id` must not be null |

All other dimension fields are null for that row.
