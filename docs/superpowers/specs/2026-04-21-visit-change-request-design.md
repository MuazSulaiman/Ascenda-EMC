# Visit Change Request Module — Design Spec
**Date:** 2026-04-21  
**Status:** Approved

---

## Context

Field reps submit visits that sometimes contain errors (wrong customer, wrong product, incorrect evaluation, etc.). Currently there is no formal way to correct a submitted visit — reps have no recourse and admins have no audit trail.

The Visit Change Request module allows reps to submit a formal request to edit their own visits. An admin reviews the request, sees a before/after diff, and can approve (auto-applying the changes to the visits table) or reject (with a note). This ensures visit data integrity while giving reps a self-service correction path.

The submission UI in `app_pages/change_request.py` is ~95% complete. What's missing is: the correct DB schema in `init_db_v11.py`, a few field-name fixes, the withdrawal feature, and the admin review page.

---

## Workflow States

```
Rep submits → IN_REVIEW → APPROVED (changes applied to visits)
                       → REJECTED  (reject_note filled, visits unchanged)
             Rep cancels → WITHDRAWN (before admin acts)
```

## Database Schema

### Tables to add to `init_db_v11.py`

**`request_changes`** — one row per change request:
```sql
CREATE TABLE IF NOT EXISTS request_changes (
    request_id    INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    visit_id      INTEGER NOT NULL REFERENCES visits(visit_id) ON DELETE CASCADE,
    requested_by  INTEGER NOT NULL REFERENCES users(user_id),
    change_source TEXT NOT NULL DEFAULT 'REQUEST',
    request_note  TEXT,
    status        TEXT NOT NULL DEFAULT 'IN_REVIEW'
                  CHECK (status IN ('IN_REVIEW','APPROVED','REJECTED','WITHDRAWN')),
    request_date  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    resolve_date  TIMESTAMPTZ,
    reject_note   TEXT,
    changed_by    INTEGER REFERENCES users(user_id),
    apply_error   TEXT,
    applied_at    TIMESTAMPTZ
);
```

**`request_change_details`** — one row per changed field:
```sql
CREATE TABLE IF NOT EXISTS request_change_details (
    detail_id   INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    request_id  INTEGER NOT NULL REFERENCES request_changes(request_id) ON DELETE CASCADE,
    field       TEXT NOT NULL,
    old_value   TEXT,
    new_value   TEXT
);
```

> Note: These tables already exist in the live DB. Adding them to `init_db_v11.py` with `IF NOT EXISTS` is safe.

## New File: `app_pages/admin_change_requests.py`

Two tabs following the `review_customers.py` / `review_audiences.py` 3-step pattern.

### Tab 1 — Review Pending

**Step 1: Summary**
- Show count badge: `"X requests pending review"` using `status_badge('IN_REVIEW', 'warning')`.
- If 0 pending: show `st.info("No pending change requests.")` and stop.

**Step 2: Select**
- Selectbox populated from:
  ```sql
  SELECT rc.request_id, rc.visit_id, u.name AS rep_name,
         rc.request_date,
         COUNT(rcd.detail_id) AS fields_changed
  FROM request_changes rc
  JOIN users u ON u.user_id = rc.requested_by
  LEFT JOIN request_change_details rcd ON rcd.request_id = rc.request_id
  WHERE rc.status = 'IN_REVIEW'
  GROUP BY rc.request_id, rc.visit_id, u.name, rc.request_date
  ORDER BY rc.request_date ASC
  ```
- Display format: `Request #14 — Visit #307 — Ahmed Khalid — Apr 18 (3 fields)`

**Step 3: Review and Resolve**
After selecting:
- **Visit context card**: customer name, visit date, rep name, business line.
- **Request note**: `st.info(f'Rep note: "{request_note}"')`.
- **Diff table**: render one `compare_row(field, old_value, new_value, changed=True)` per row in `request_change_details` for this request_id.
- **Two action columns (col1 | col2)**:

  *Left — Approve:*
  ```
  [Approve & Apply Changes]  ← primary button
  ```
  On click:
  1. Open `engine.begin()` transaction:
     - For each `(field, new_value)` in `request_change_details` (whitelist-checked):
       ```sql
       UPDATE visits SET {field} = :val WHERE visit_id = :vid
       ```
     - `UPDATE request_changes SET status='APPROVED', applied_at=NOW(), changed_by=:admin_uid WHERE request_id=:rid`
  2. If the transaction raises an exception (rolls back automatically):
     - Open a **separate** connection (outside the failed transaction) to record the error:
       ```sql
       UPDATE request_changes SET apply_error=:err WHERE request_id=:rid
       ```
     - Status remains `IN_REVIEW` (not changed by the failed transaction).
     - Show `st.error(f"Apply failed: {e}")` to the admin.

  *Right — Reject:*
  ```
  Rejection reason: [text_area]
  [Reject Request]           ← danger-styled button
  ```
  On click:
  ```sql
  UPDATE request_changes
  SET status='REJECTED', reject_note=:note, resolve_date=NOW(), changed_by=:admin_uid
  WHERE request_id=:rid
  ```
  Does NOT touch the visits table.

- Both actions: show success message (session state pattern), then `st.rerun()`.

## Auto-Apply Field Mapping

When applying `request_change_details` fields to `visits`, the `field` column contains the actual `visits` column name (e.g., `customer_id`, `notes`, `evaluation`, `audience_id`, `product_id`, `objective_id`, `business_line_id`).

The UPDATE is dynamic:
```python
for row in change_details:
    conn.execute(
        text(f"UPDATE visits SET {row.field} = :val WHERE visit_id = :vid"),
        {"val": row.new_value, "vid": visit_id}
    )
```

**Allowed field whitelist** (guard before dynamic UPDATE):
```python
ALLOWED_VISIT_FIELDS = {
    'customer_id', 'audience_id', 'business_line_id', 'product_id',
    'objective_id', 'notes', 'evaluation', 'project_id',
    'other_customer_name', 'other_audience_title', 'other_audience_name',
    'other_audience_department', 'other_audience_position',
    'other_audience_phone', 'other_audience_email',
}
```
If any `field` is not in this set, abort the transaction and surface an error — do not apply any changes.
